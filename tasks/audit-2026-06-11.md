# ClipStory Audit — Bugs & Mac↔iOS Parity Gaps (2026-06-11)

Multi-agent audit: 53 agents, 43 raw findings → 38 confirmed after adversarial verification, 5 refuted.
Note: the working-tree changes present at audit start were committed mid-audit as `b455fd7` "Polish iOS copy and sync UX" — findings marked (b455fd7) refer to that changeset.

---

## TL;DR — why your three symptoms happen

All three symptoms ("pages missing on iOS", "clips missing on iOS", "locked pinboard from Mac not on iOS") share one root cause plus a few real data-loss bugs on top:

1. **Your Mac and iPhone are talking to two different CloudKit databases.** The Sparkle-distributed Mac app is a Developer ID **Release** build → CloudKit **Production** environment (forced by `scripts/exportOptions.developer-id.plist` since the first automated release; Developer ID builds *cannot* use Development). Your iPhone runs an Xcode **Debug** build → CloudKit **Development** environment. These are disjoint databases — nothing created in the daily Mac app can ever reach the phone. Anything that *did* sync was created while you happened to run a Mac Debug build, which is exactly why only *some* pages/clips are missing.
2. **The CloudKit Production schema was never deployed** (`docs/DISTRIBUTION.md:30`, unchecked checklist item at `:197`). Either it was truly never deployed (→ ALL Release-build sync has been dead since 1.0.0), or it was deployed before 1.0.10 and is now missing the new `lockSalt`/`lockHash` fields added in `c86d43a` (→ every Pinboard touched on a 1.0.10+ Release build fails export, and the failing record stalls the zone's export pipeline, taking subsequent clips/pages down with it).
3. **You can't see any of this** because sync status only checks `CKContainer.accountStatus()` — both devices say "iCloud sync is on" while exports fail silently. The new "Refresh iCloud" button and pull-to-refresh (b455fd7) are placebos: they re-check account status and run a local dedupe sweep, but never trigger a CloudKit import.

**The locked pinboard itself is stored correctly** — `lockSalt`/`lockHash` are synced SwiftData attributes on `Pinboard` (`Shared/Models/Pinboard.swift:14-15`), not Keychain/UserDefaults. It's the transport that's broken.

### 15-minute diagnosis to settle it
- CloudKit Console → `iCloud.com.vladbabic.clipstory` → Schema (**Production**): does `CD_Pinboard` exist, and does it have `CD_lockSalt`/`CD_lockHash`?
- Both Settings screens now show an **Environment** row (b455fd7) — confirm Mac says Production and iPhone says Development (the mismatch).
- On the Release Mac: `log stream --predicate 'subsystem CONTAINS "coredata"'` and copy something — look for CKError partial-failure / invalid-arguments on export.

### Fix order (sync)
1. Deploy the Development schema to Production in CloudKit Console (and re-do this after ANY `@Model` attribute change — add to release checklist).
2. Get the iPhone onto a matching environment: upload the iOS build to TestFlight (unchecked at `DISTRIBUTION.md:208`) so both run Production — or test Mac Debug + iOS Debug.
3. Fix `macOS/ClipStory.entitlements:15-16`: `aps-environment` is hardcoded `development`; Developer ID builds need `production` (parameterize like `ICLOUD_CONTAINER_ENVIRONMENT`). Without it the Mac gets no CloudKit pushes and only refreshes opportunistically.
4. Surface `NSPersistentCloudKitContainer.eventChangedNotification` errors in `CloudSyncStatus` so export failures are visible instead of "sync is on".

## Critical & high bugs

### Data loss

| # | Finding | Where |
|---|---------|-------|
| C1 | **Prune treats `pinboard == nil && !isPinned` as disposable.** CloudKit imports ClipItem records before their Pinboard relationship links (initial sync, new pinboard, restored device). In that window, page-assigned clips look prunable; any insert under a synced history limit deletes them, and the delete syncs back — permanent, cross-device loss of page clips. Mac-side inserts (every copy) trigger prune unconditionally. | `ClipStore.swift:207-255` |
| C2 | **`dedupeSweep` races in-flight imports.** Runs at iOS foreground and now on pull-to-refresh (b455fd7) — exactly when imports are mid-flight. Drops the duplicate's pinboard membership (only fills a nil keeper), never merges `customTitle`/`recognizedText`, loses Mac-fetched URL/file previews, and the survivor becomes prunable → feeds C1. | `ClipStore+Dedupe.swift:13-42` |
| C3 | **iOS share extension strands clips.** No app group — the extension writes to its own sandboxed `default.store`, then `completeRequest` fires ~0.65s after save, killing the process before the async CloudKit export finishes. Shared clips usually never reach the iOS app or the Mac (occasionally arrive late via a retry on a later share). | `ShareViewController.swift:67-97`, `ModelContainerFactory` |
| C4 | **Save errors swallowed without rollback** — failed saves leave the context dirty; the orphaned change commits as a side effect of the next unrelated save or is lost on quit. | `ClipStore.swift:271-277` |

### Locked-pinboard security

| # | Finding | Where |
|---|---------|-------|
| S1 | **Face ID bypasses the password entirely** (b455fd7). Biometric success inserts the board into `unlockedBoardIDs` without ever calling `PinboardLocking.verify` — any enrolled face/finger on the device opens every Mac-locked page. Auto-fires on appear. Fix: only offer biometrics after one successful password entry on that device (1Password pattern). | `HistoryListView.swift:1092-1121` |
| S2 | **Lock is view-layer only; clips sync in plaintext.** A pre-1.0.10 iOS build has no lock fields — CloudKit import ignores unknown fields and the page renders fully open, no password, no lock icon. The code comment admits "does not encrypt"; the UI never does. | `PinboardLocking.swift:4-6`, `Pinboard.swift:38-40` |
| S3 | **Weak KDF, synced hash.** 25k iterations of raw SHA-256 (<1ms), 4-char minimum, hash synced to CloudKit → offline crack of the user's (likely reused) password in seconds for anyone with DB/record access. `isLocked` also fails open if only one of salt/hash is set. | `PinboardLocking.swift:8-33` |
| S4 | Source-app filter menu leaks app names of clips inside still-locked pages (iOS). | `HistoryListView.swift:96-102` |
| S5 | Add-to-Page menus on both platforms let you file a clip into a locked page without unlocking — it instantly vanishes from history/search; locked clips still inflate achievements/stats. | both platforms |

### Sync UX

- **"Refresh iCloud" + pull-to-refresh never import** — they check account status and dedupe locally; success haptic + "Checked 12:34" timestamp falsely imply data is current. (`HistoryListView.swift:405-448, 929-933`)
- **Sync status = account status.** No `eventChangedNotification` observer; persistent export failures produce zero signal. (`CloudSyncStatus.swift:70-80`)
- **iOS silent local-only fallback** on container failure (realistically simulator/unsigned builds only — `EntitlementChecker` returns true unconditionally on device). (`ModelContainerFactory.swift:34-61`)

### Medium/low (selection)

- Mac drops the **entire** clip (text included) when a text+image pasteboard's image fails re-encode under the 8 MB cap. (`ClipboardMonitor.swift:254-315`)
- iOS auto-import marks the pasteboard generation consumed even when `store.insert` fails → that content never auto-imports again. (`ClipStoryiOSApp.swift:65-71`)
- URL clips paste only a plain string (no `public.url` flavor); `updateText` has a no-op self-assignment. (`PasteService.swift:55-63`, `ClipStore.swift:171`)
- `dedupeSweep` collapses the same content deliberately saved to two pages → vanishes from one page on every device.
- Data written to the wrong environment has no migration path (records are marked "exported" and won't re-export after switching environments).

## Mac ↔ iOS parity gaps (prioritized)

1. **iOS can't rename or delete pinboards.** The only rename/delete UI (`PinboardChipsRow.swift`) is dead code — never instantiated. Every iOS-created page is permanently "Untitled" and undeletable without a Mac. Store methods already exist; only UI wiring is missing. *(This is your "missing pages" adjacent gap — also note: no iOS code path hides pinboards; locked boards ARE listed.)*
2. **No link previews for URLs saved on iOS** — no LinkPresentation code in iOS/ or Shared/; Mac only fetches at capture time. iOS-saved URLs show a bare icon forever, on all devices.
3. **Share-extension images skip OCR** and nothing backfills → screenshots shared on iOS (a primary capture path) are permanently unsearchable.
4. **Mac has no Pin/Unpin action** (zero `togglePin` references in macOS UI) despite a "Pinned Only" filter and pin-aware pruning — Mac users can't protect a clip from pruning; iPhone pins can't be toggled on Mac.
5. **Mac file clips degrade silently on iOS** — no `fileData` synced; grid-tap Copy silently yields a preview image or filename string.
6. **JSON export is Mac-only; import exists nowhere** — an iOS-only user can't get data out.
7. **Mac unlock is password-only** — no Touch ID via LAContext (iOS has biometrics; asymmetric the other way).
8. **macOS Progress pane shows only 4 of 8 achievements** — `achievements.prefix(4)` at `SettingsView.swift:416`; looks like a truncation bug.
9. **No iOS quick-access surfaces** — no widget, keyboard extension, or lock-screen/Control Center entry; only Siri App Shortcuts. Main ceiling on iOS usefulness.
10. **iOS root `@Query` fetches every ClipItem** with no fetchLimit/predicate; all filtering in-memory — risk at the 2000-item limit on older iPhones.
11. Shared gaps vs Paste: no page reordering on either platform; no RTF/attributed-text capture; Mac lacks clip detail view / rotate / multi-select; hotkey hard-coded Shift+Cmd+V.

Capture-side differences that are OS limitations, not bugs: iOS can't background-read the pasteboard, can't know the source app, can't paste back into other apps.

## Refuted by verification (don't chase these)

- iOS provisioning profiles "swapped/missing" — config is fine.
- "Locked-board clips hidden from iOS main history / board relocks per appear" — behavior matches Mac.
- "CloudKit image assets render as broken placeholders" — external storage handled correctly.

## Audit blind spots (flagged by completeness critic)

- CloudKit Console state itself is unverifiable from the repo — run the 15-minute diagnosis above.
- `Tests/` covers single-store logic only; the prune/dedupe **sync-race** bugs (C1/C2) have zero regression coverage.
- `resolvedSettings()` mutates/deletes duplicate AppSettings on a read path — same mutate-on-read pattern as dedupeSweep; benign today, worth watching.
- `deletePinboard` nullifies membership → freed clips instantly become prunable under a synced limit.

## Suggested attack order

1. Schema deploy + environment alignment + macOS `aps-environment` (unblocks everything; 1 hour).
2. C1 prune race (don't treat possibly-unlinked clips as disposable — e.g., minimum age since import, or exclude items with pending relationship) + make dedupe merge `customTitle`/`recognizedText`/previews and skip while import in flight.
3. S1 Face ID bypass.
4. Real sync-error surfacing (event notifications) + make "Refresh iCloud" honest.
5. App group + shared store for the share extension.
6. Wire up iOS rename/delete page (resurrect `PinboardChipsRow` logic into the live pagePicker).
